45  /**
46   * A hash table supporting full concurrency of retrievals and
47   * high expected concurrency for updates. This class obeys the
48   * same functional specification as {@link java.util.Hashtable}, and
49   * includes versions of methods corresponding to each method of
50   * {@code Hashtable}. However, even though all operations are
51   * thread-safe, retrieval operations do <em>not</em> entail locking,
52   * and there is <em>not</em> any support for locking the entire table
53   * in a way that prevents all access.  This class is fully
54   * interoperable with {@code Hashtable} in programs that rely on its
55   * thread safety but not on its synchronization details.
56   *
57   * <p>Retrieval operations (including {@code get}) generally do not
58   * block, so may overlap with update operations (including {@code put}
59   * and {@code remove}). Retrievals reflect the results of the most
60   * recently <em>completed</em> update operations holding upon their
61   * onset. (More formally, an update operation for a given key bears a
62   * <em>happens-before</em> relation with any (non-null) retrieval for
63   * that key reporting the updated value.)  For aggregate operations
64   * such as {@code putAll} and {@code clear}, concurrent retrievals may
65   * reflect insertion or removal of only some entries.  Similarly,
66   * Iterators and Enumerations return elements reflecting the state of
67   * the hash table at some point at or since the creation of the
68   * iterator/enumeration.  They do <em>not</em> throw {@link
69   * ConcurrentModificationException}.  However, iterators are designed
70   * to be used by only one thread at a time.  Bear in mind that the
71   * results of aggregate status methods including {@code size}, {@code
72   * isEmpty}, and {@code containsValue} are typically useful only when
73   * a map is not undergoing concurrent updates in other threads.
74   * Otherwise the results of these methods reflect transient states
75   * that may be adequate for monitoring or estimation purposes, but not
76   * for program control.
77   *
78   * <p>The table is dynamically expanded when there are too many
79   * collisions (i.e., keys that have distinct hash codes but fall into
80   * the same slot modulo the table size), with the expected average
81   * effect of maintaining roughly two bins per mapping (corresponding
82   * to a 0.75 load factor threshold for resizing). There may be much
83   * variance around this average as mappings are added and removed, but
84   * overall, this maintains a commonly accepted time/space tradeoff for
85   * hash tables.  However, resizing this or any other kind of hash
86   * table may be a relatively slow operation. When possible, it is a
87   * good idea to provide a size estimate as an optional {@code
88   * initialCapacity} constructor argument. An additional optional
89   * {@code loadFactor} constructor argument provides a further means of
90   * customizing initial table capacity by specifying the table density
91   * to be used in calculating the amount of space to allocate for the
92   * given number of elements.  Also, for compatibility with previous
93   * versions of this class, constructors may optionally specify an
94   * expected {@code concurrencyLevel} as an additional hint for
95   * internal sizing.  Note that using many keys with exactly the same
96   * {@code hashCode()} is a sure way to slow down performance of any
97   * hash table. To ameliorate impact, when keys are {@link Comparable},
98   * this class may use comparison order among keys to help break ties.
99   *
100  * <p>A {@link Set} projection of a ConcurrentHashMapV8 may be created
101  * (using {@link #newKeySet()} or {@link #newKeySet(int)}), or viewed
102  * (using {@link #keySet(Object)} when only keys are of interest, and the
103  * mapped values are (perhaps transiently) not used or all take the
104  * same mapping value.
105  *
106  * <p>This class and its views and iterators implement all of the
107  * <em>optional</em> methods of the {@link Map} and {@link Iterator}
108  * interfaces.
109  *
110  * <p>Like {@link Hashtable} but unlike {@link HashMap}, this class
111  * does <em>not</em> allow {@code null} to be used as a key or value.
112  *
113  * <p>ConcurrentHashMapV8s support a set of sequential and parallel bulk
114  * operations that are designed
115  * to be safely, and often sensibly, applied even with maps that are
116  * being concurrently updated by other threads; for example, when
117  * computing a snapshot summary of the values in a shared registry.
118  * There are three kinds of operation, each with four forms, accepting
119  * functions with Keys, Values, Entries, and (Key, Value) arguments
120  * and/or return values. Because the elements of a ConcurrentHashMapV8
121  * are not ordered in any particular way, and may be processed in
122  * different orders in different parallel executions, the correctness
123  * of supplied functions should not depend on any ordering, or on any
124  * other objects or values that may transiently change while
125  * computation is in progress; and except for forEach actions, should
126  * ideally be side-effect-free. Bulk operations on {@link java.util.Map.Entry}
127  * objects do not support method {@code setValue}.
128  *
129  * <ul>
130  * <li> forEach: Perform a given action on each element.
131  * A variant form applies a given transformation on each element
132  * before performing the action.</li>
133  *
134  * <li> search: Return the first available non-null result of
135  * applying a given function on each element; skipping further
136  * search when a result is found.</li>
137  *
138  * <li> reduce: Accumulate each element.  The supplied reduction
139  * function cannot rely on ordering (more formally, it should be
140  * both associative and commutative).  There are five variants:
141  *
142  * <ul>
143  *
144  * <li> Plain reductions. (There is not a form of this method for
145  * (key, value) function arguments since there is no corresponding
146  * return type.)</li>
147  *
148  * <li> Mapped reductions that accumulate the results of a given
149  * function applied to each element.</li>
150  *
151  * <li> Reductions to scalar doubles, longs, and ints, using a
152  * given basis value.</li>
153  *
154  * </ul>
155  * </li>
156  * </ul>
157  *
158  * <p>These bulk operations accept a {@code parallelismThreshold}
159  * argument. Methods proceed sequentially if the current map size is
160  * estimated to be less than the given threshold. Using a value of
161  * {@code Long.MAX_VALUE} suppresses all parallelism.  Using a value
162  * of {@code 1} results in maximal parallelism by partitioning into
163  * enough subtasks to fully utilize the {@link
164  * ForkJoinPool#commonPool()} that is used for all parallel
165  * computations. Normally, you would initially choose one of these
166  * extreme values, and then measure performance of using in-between
167  * values that trade off overhead versus throughput.
168  *
169  * <p>The concurrency properties of bulk operations follow
170  * from those of ConcurrentHashMapV8: Any non-null result returned
171  * from {@code get(key)} and related access methods bears a
172  * happens-before relation with the associated insertion or
173  * update.  The result of any bulk operation reflects the
174  * composition of these per-element relations (but is not
175  * necessarily atomic with respect to the map as a whole unless it
176  * is somehow known to be quiescent).  Conversely, because keys
177  * and values in the map are never null, null serves as a reliable
178  * atomic indicator of the current lack of any result.  To
179  * maintain this property, null serves as an implicit basis for
180  * all non-scalar reduction operations. For the double, long, and
181  * int versions, the basis should be one that, when combined with
182  * any other value, returns that other value (more formally, it
183  * should be the identity element for the reduction). Most common
184  * reductions have these properties; for example, computing a sum
185  * with basis 0 or a minimum with basis MAX_VALUE.
186  *
187  * <p>Search and transformation functions provided as arguments
188  * should similarly return null to indicate the lack of any result
189  * (in which case it is not used). In the case of mapped
190  * reductions, this also enables transformations to serve as
191  * filters, returning null (or, in the case of primitive
192  * specializations, the identity basis) if the element should not
193  * be combined. You can create compound transformations and
194  * filterings by composing them yourself under this "null means
195  * there is nothing there now" rule before using them in search or
196  * reduce operations.
197  *
198  * <p>Methods accepting and/or returning Entry arguments maintain
199  * key-value associations. They may be useful for example when
200  * finding the key for the greatest value. Note that "plain" Entry
201  * arguments can be supplied using {@code new
202  * AbstractMap.SimpleEntry(k,v)}.
203  *
204  * <p>Bulk operations may complete abruptly, throwing an
205  * exception encountered in the application of a supplied
206  * function. Bear in mind when handling such exceptions that other
207  * concurrently executing functions could also have thrown
208  * exceptions, or would have done so if the first exception had
209  * not occurred.
210  *
211  * <p>Speedups for parallel compared to sequential forms are common
212  * but not guaranteed.  Parallel operations involving brief functions
213  * on small maps may execute more slowly than sequential forms if the
214  * underlying work to parallelize the computation is more expensive
215  * than the computation itself.  Similarly, parallelization may not
216  * lead to much actual parallelism if all processors are busy
217  * performing unrelated tasks.
218  *
219  * <p>All arguments to all task methods must be non-null.
220  *
221  * <p><em>jsr166e note: During transition, this class
222  * uses nested functional interfaces with different names but the
223  * same forms as those expected for JDK8.</em>
224  *
225  * <p>This class is a member of the
226  * <a href="{@docRoot}/../technotes/guides/collections/index.html">
227  * Java Collections Framework</a>.
228  *
229  * @since 1.5
230  * @author Doug Lea
231  * @param <K> the type of keys maintained by this map
232  * @param <V> the type of mapped values
233  */

292     /*
293      * Overview:
294      *
295      * The primary design goal of this hash table is to maintain
296      * concurrent readability (typically method get(), but also
297      * iterators and related methods) while minimizing update
298      * contention. Secondary goals are to keep space consumption about
299      * the same or better than java.util.HashMap, and to support high
300      * initial insertion rates on an empty table by many threads.
301      *
302      * This map usually acts as a binned (bucketed) hash table.  Each
303      * key-value mapping is held in a Node.  Most nodes are instances
304      * of the basic Node class with hash, key, value, and next
305      * fields. However, various subclasses exist: TreeNodes are
306      * arranged in balanced trees, not lists.  TreeBins hold the roots
307      * of sets of TreeNodes. ForwardingNodes are placed at the heads
308      * of bins during resizing. ReservationNodes are used as
309      * placeholders while establishing values in computeIfAbsent and
310      * related methods.  The types TreeBin, ForwardingNode, and
311      * ReservationNode do not hold normal user keys, values, or
312      * hashes, and are readily distinguishable during search etc
313      * because they have negative hash fields and null key and value
314      * fields. (These special nodes are either uncommon or transient,
315      * so the impact of carrying around some unused fields is
316      * insignificant.)
317      *
318      * The table is lazily initialized to a power-of-two size upon the
319      * first insertion.  Each bin in the table normally contains a
320      * list of Nodes (most often, the list has only zero or one Node).
321      * Table accesses require volatile/atomic reads, writes, and
322      * CASes.  Because there is no other way to arrange this without
323      * adding further indirections, we use intrinsics
324      * (sun.misc.Unsafe) operations.
325      *
326      * We use the top (sign) bit of Node hash fields for control
327      * purposes -- it is available anyway because of addressing
328      * constraints.  Nodes with negative hash fields are specially
329      * handled or ignored in map methods.
330      *
331      * Insertion (via put or its variants) of the first node in an
332      * empty bin is performed by just CASing it to the bin.  This is
333      * by far the most common case for put operations under most
334      * key/hash distributions.  Other update operations (insert,
335      * delete, and replace) require locks.  We do not want to waste
336      * the space required to associate a distinct lock object with
337      * each bin, so instead use the first node of a bin list itself as
338      * a lock. Locking support for these locks relies on builtin
339      * "synchronized" monitors.
340      *
341      * Using the first node of a list as a lock does not by itself
342      * suffice though: When a node is locked, any update must first
343      * validate that it is still the first node after locking it, and
344      * retry if not. Because new nodes are always appended to lists,
345      * once a node is first in a bin, it remains first until deleted
346      * or the bin becomes invalidated (upon resizing).
347      *
348      * The main disadvantage of per-bin locks is that other update
349      * operations on other nodes in a bin list protected by the same
350      * lock can stall, for example when user equals() or mapping
351      * functions take a long time.  However, statistically, under
352      * random hash codes, this is not a common problem.  Ideally, the
353      * frequency of nodes in bins follows a Poisson distribution
354      * (http://en.wikipedia.org/wiki/Poisson_distribution) with a
355      * parameter of about 0.5 on average, given the resizing threshold
356      * of 0.75, although with a large variance because of resizing
357      * granularity. Ignoring variance, the expected occurrences of
358      * list size k are (exp(-0.5) * pow(0.5, k) / factorial(k)). The
359      * first values are:
360      *
361      * 0:    0.60653066
362      * 1:    0.30326533
363      * 2:    0.07581633
364      * 3:    0.01263606
365      * 4:    0.00157952
366      * 5:    0.00015795
367      * 6:    0.00001316
368      * 7:    0.00000094
369      * 8:    0.00000006
370      * more: less than 1 in ten million
371      *
372      * Lock contention probability for two threads accessing distinct
373      * elements is roughly 1 / (8 * #elements) under random hashes.
374      *
375      * Actual hash code distributions encountered in practice
376      * sometimes deviate significantly from uniform randomness.  This
377      * includes the case when N > (1<<30), so some keys MUST collide.
378      * Similarly for dumb or hostile usages in which multiple keys are
379      * designed to have identical hash codes or ones that differs only
380      * in masked-out high bits. So we use a secondary strategy that
381      * applies when the number of nodes in a bin exceeds a
382      * threshold. These TreeBins use a balanced tree to hold nodes (a
383      * specialized form of red-black trees), bounding search time to
384      * O(log N).  Each search step in a TreeBin is at least twice as
385      * slow as in a regular list, but given that N cannot exceed
386      * (1<<64) (before running out of addresses) this bounds search
387      * steps, lock hold times, etc, to reasonable constants (roughly
388      * 100 nodes inspected per operation worst case) so long as keys
389      * are Comparable (which is very common -- String, Long, etc).
390      * TreeBin nodes (TreeNodes) also maintain the same "next"
391      * traversal pointers as regular nodes, so can be traversed in
392      * iterators in the same way.
393      *
394      * The table is resized when occupancy exceeds a percentage
395      * threshold (nominally, 0.75, but see below).  Any thread
396      * noticing an overfull bin may assist in resizing after the
397      * initiating thread allocates and sets up the replacement
398      * array. However, rather than stalling, these other threads may
399      * proceed with insertions etc.  The use of TreeBins shields us
400      * from the worst case effects of overfilling while resizes are in
401      * progress.  Resizing proceeds by transferring bins, one by one,
402      * from the table to the next table. To enable concurrency, the
403      * next table must be (incrementally) prefilled with place-holders
404      * serving as reverse forwarders to the old table.  Because we are
405      * using power-of-two expansion, the elements from each bin must
406      * either stay at same index, or move with a power of two
407      * offset. We eliminate unnecessary node creation by catching
408      * cases where old nodes can be reused because their next fields
409      * won't change.  On average, only about one-sixth of them need
410      * cloning when a table doubles. The nodes they replace will be
411      * garbage collectable as soon as they are no longer referenced by
412      * any reader thread that may be in the midst of concurrently
413      * traversing table.  Upon transfer, the old table bin contains
414      * only a special forwarding node (with hash field "MOVED") that
415      * contains the next table as its key. On encountering a
416      * forwarding node, access and update operations restart, using
417      * the new table.
418      *
419      * Each bin transfer requires its bin lock, which can stall
420      * waiting for locks while resizing. However, because other
421      * threads can join in and help resize rather than contend for
422      * locks, average aggregate waits become shorter as resizing
423      * progresses.  The transfer operation must also ensure that all
424      * accessible bins in both the old and new table are usable by any
425      * traversal.  This is arranged by proceeding from the last bin
426      * (table.length - 1) up towards the first.  Upon seeing a
427      * forwarding node, traversals (see class Traverser) arrange to
428      * move to the new table without revisiting nodes.  However, to
429      * ensure that no intervening nodes are skipped, bin splitting can
430      * only begin after the associated reverse-forwarders are in
431      * place.
432      *
433      * The traversal scheme also applies to partial traversals of
434      * ranges of bins (via an alternate Traverser constructor)
435      * to support partitioned aggregate operations.  Also, read-only
436      * operations give up if ever forwarded to a null table, which
437      * provides support for shutdown-style clearing, which is also not
438      * currently implemented.
439      *
440      * Lazy table initialization minimizes footprint until first use,
441      * and also avoids resizings when the first operation is from a
442      * putAll, constructor with map argument, or deserialization.
443      * These cases attempt to override the initial capacity settings,
444      * but harmlessly fail to take effect in cases of races.
445      *
446      * The element count is maintained using a specialization of
447      * LongAdder. We need to incorporate a specialization rather than
448      * just use a LongAdder in order to access implicit
449      * contention-sensing that leads to creation of multiple
450      * CounterCells.  The counter mechanics avoid contention on
451      * updates but can encounter cache thrashing if read too
452      * frequently during concurrent access. To avoid reading so often,
453      * resizing under contention is attempted only upon adding to a
454      * bin already holding two or more nodes. Under uniform hash
455      * distributions, the probability of this occurring at threshold
456      * is around 13%, meaning that only about 1 in 8 puts check
457      * threshold (and after resizing, many fewer do so).
458      *
459      * TreeBins use a special form of comparison for search and
460      * related operations (which is the main reason we cannot use
461      * existing collections such as TreeMaps). TreeBins contain
462      * Comparable elements, but may contain others, as well as
463      * elements that are Comparable but not necessarily Comparable
464      * for the same T, so we cannot invoke compareTo among them. To
465      * handle this, the tree is ordered primarily by hash value, then
466      * by Comparable.compareTo order if applicable.  On lookup at a
467      * node, if elements are not comparable or compare as 0 then both
468      * left and right children may need to be searched in the case of
469      * tied hash values. (This corresponds to the full list search
470      * that would be necessary if all elements were non-Comparable and
471      * had tied hashes.)  The red-black balancing code is updated from
472      * pre-jdk-collections
473      * (http://gee.cs.oswego.edu/dl/classes/collections/RBCell.java)
474      * based in turn on Cormen, Leiserson, and Rivest "Introduction to
475      * Algorithms" (CLR).
476      *
477      * TreeBins also require an additional locking mechanism.  While
478      * list traversal is always possible by readers even during
479      * updates, tree traversal is not, mainly because of tree-rotations
480      * that may change the root node and/or its linkages.  TreeBins
481      * include a simple read-write lock mechanism parasitic on the
482      * main bin-synchronization strategy: Structural adjustments
483      * associated with an insertion or removal are already bin-locked
484      * (and so cannot conflict with other writers) but must wait for
485      * ongoing readers to finish. Since there can be only one such
486      * waiter, we use a simple scheme using a single "waiter" field to
487      * block writers.  However, readers need never block.  If the root
488      * lock is held, they proceed along the slow traversal path (via
489      * next-pointers) until the lock becomes available or the list is
490      * exhausted, whichever comes first. These cases are not fast, but
491      * maximize aggregate expected throughput.
492      *
493      * Maintaining API and serialization compatibility with previous
494      * versions of this class introduces several oddities. Mainly: We
495      * leave untouched but unused constructor arguments refering to
496      * concurrencyLevel. We accept a loadFactor constructor argument,
497      * but apply it only to initial table capacity (which is the only
498      * time that we can guarantee to honor it.) We also declare an
499      * unused "Segment" class that is instantiated in minimal form
500      * only when serializing.
501      *
502      * This file is organized to make things a little easier to follow
503      * while reading than they might otherwise: First the main static
504      * declarations and utilities, then fields, then main public
505      * methods (with a few factorings of multiple public methods into
506      * internal ones), then sizing methods, trees, traversers, and
507      * bulk operations.
508      */

659     /**
660      * Spreads (XORs) higher bits of hash to lower and also forces top
661      * bit to 0. Because the table uses power-of-two masking, sets of
662      * hashes that vary only in bits above the current mask will
663      * always collide. (Among known examples are sets of Float keys
664      * holding consecutive whole numbers in small tables.)  So we
665      * apply a transform that spreads the impact of higher bits
666      * downward. There is a tradeoff between speed, utility, and
667      * quality of bit-spreading. Because many common sets of hashes
668      * are already reasonably distributed (so don't benefit from
669      * spreading), and because we use trees to handle large sets of
670      * collisions in bins, we just XOR some shifted bits in the
671      * cheapest possible way to reduce systematic lossage, as well as
672      * to incorporate impact of the highest bits that would otherwise
673      * never be used in index calculations because of table bounds.
674      */
675     static final int spread(int h) {
676         return (h ^ (h >>> 16)) & HASH_BITS;
677     }

681      * See Hackers Delight, sec 3.2

1005     /** Implementation for put and putIfAbsent */
1006     final V putVal(K key, V value, boolean onlyIfAbsent) {
1007         if (key == null || value == null) throw new NullPointerException();
1008         int hash = spread(key.hashCode());
1009         int binCount = 0;
1010         for (Node<K,V>[] tab = table;;) {
1011             Node<K,V> f; int n, i, fh;
1012             if (tab == null || (n = tab.length) == 0)
1013                 tab = initTable();
1014             else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
1015                 if (casTabAt(tab, i, null,
1016                         new Node<K,V>(hash, key, value, null)))
1017                     break;                   // no lock when adding to empty bin
1018             }
1019             else if ((fh = f.hash) == MOVED)
1020                 tab = helpTransfer(tab, f);
1021             else {
1022                 V oldVal = null;
1023                 synchronized (f) {
1024                     if (tabAt(tab, i) == f) {
1025                         if (fh >= 0) {
1026                             binCount = 1;
1027                             for (Node<K,V> e = f;; ++binCount) {
1028                                 K ek;
1029                                 if (e.hash == hash &&
1030                                         ((ek = e.key) == key ||
1031                                                 (ek != null && key.equals(ek)))) {
1032                                     oldVal = e.val;
1033                                     if (!onlyIfAbsent)
1034                                         e.val = value;
1035                                     break;
1036                                 }
1037                                 Node<K,V> pred = e;
1038                                 if ((e = e.next) == null) {
1039                                     pred.next = new Node<K,V>(hash, key,
1040                                             value, null);
1041                                     break;
1042                                 }
1043                             }
1044                         }
1045                         else if (f instanceof TreeBin) {
1046                             Node<K,V> p;
1047                             binCount = 2;
1048                             if ((p = ((TreeBin<K,V>)f).putTreeVal(hash, key,
1049                                     value)) != null) {
1050                                 oldVal = p.val;
1051                                 if (!onlyIfAbsent)
1052                                     p.val = value;
1053                             }
1054                         }
1055                     }
1056                 }
1057                 if (binCount != 0) {
1058                     if (binCount >= TREEIFY_THRESHOLD)
1059                         treeifyBin(tab, i);
1060                     if (oldVal != null)
1061                         return oldVal;
1062                     break;
1063                 }
1064             }
1065         }
1066         addCount(1L, binCount);
1067         return null;
1068     }

1098      * Replaces node value with v, conditional upon match of cv if
1099      * non-null.  If resulting value is null, delete.
1100      */
1101     final V replaceNode(Object key, V value, Object cv) {
1102         int hash = spread(key.hashCode());
1103         for (Node<K,V>[] tab = table;;) {
1104             Node<K,V> f; int n, i, fh;
1105             if (tab == null || (n = tab.length) == 0 ||
1106                     (f = tabAt(tab, i = (n - 1) & hash)) == null)
1107                 break;
1108             else if ((fh = f.hash) == MOVED)
1109                 tab = helpTransfer(tab, f);
1110             else {
1111                 V oldVal = null;
1112                 boolean validated = false;
1113                 synchronized (f) {
1114                     if (tabAt(tab, i) == f) {
1115                         if (fh >= 0) {
1116                             validated = true;
1117                             for (Node<K,V> e = f, pred = null;;) {
1118                                 K ek;
1119                                 if (e.hash == hash &&
1120                                         ((ek = e.key) == key ||
1121                                                 (ek != null && key.equals(ek)))) {
1122                                     V ev = e.val;
1123                                     if (cv == null || cv == ev ||
1124                                             (ev != null && cv.equals(ev))) {
1125                                         oldVal = ev;
1126                                         if (value != null)
1127                                             e.val = value;
1128                                         else if (pred != null)
1129                                             pred.next = e.next;
1130                                         else
1131                                             setTabAt(tab, i, e.next);
1132                                     }
1133                                     break;
1134                                 }
1135                                 pred = e;
1136                                 if ((e = e.next) == null)
1137                                     break;
1138                             }
1139                         }
1140                         else if (f instanceof TreeBin) {
1141                             validated = true;
1142                             TreeBin<K,V> t = (TreeBin<K,V>)f;
1143                             TreeNode<K,V> r, p;
1144                             if ((r = t.root) != null &&
1145                                     (p = r.findTreeNode(hash, key, null)) != null) {
1146                                 V pv = p.val;
1147                                 if (cv == null || cv == pv ||
1148                                         (pv != null && cv.equals(pv))) {
1149                                     oldVal = pv;
1150                                     if (value != null)
1151                                         p.val = value;
1152                                     else if (t.removeTreeNode(p))
1153                                         setTabAt(tab, i, untreeify(t.first));
1154                                 }
1155                             }
1156                         }
1157                     }
1158                 }
1159                 if (validated) {
1160                     if (oldVal != null) {
1161                         if (value == null)
1162                             addCount(-1L, -1);
1163                         return oldVal;
1164                     }
1165                     break;
1166                 }
1167             }
1168         }
1169         return null;
1170     }

1172     /**
1173      * Removes all of the mappings from this map.
1174      */
1175     public void clear() {
1176         long delta = 0L; // negative number of deletions
1177         int i = 0;
1178         Node<K,V>[] tab = table;
1179         while (tab != null && i < tab.length) {
1180             int fh;
1181             Node<K,V> f = tabAt(tab, i);
1182             if (f == null)
1183                 ++i;
1184             else if ((fh = f.hash) == MOVED) {
1185                 tab = helpTransfer(tab, f);
1186                 i = 0; // restart
1187             }
1188             else {
1189                 synchronized (f) {
1190                     if (tabAt(tab, i) == f) {
1191                         Node<K,V> p = (fh >= 0 ? f :
1192                                 (f instanceof TreeBin) ?
1193                                         ((TreeBin<K,V>)f).first : null);

2335                 sizeCtl = Integer.MAX_VALUE;

2340             transferIndex = n;

2342             for (int k = n; k > 0;) {    // progressively reveal ready slots
2343                 int nextk = (k > stride) ? k - stride : 0;
2344                 for (int m = nextk; m < k; ++m)
2345                     nextTab[m] = rev;

2361                     i = -1;

2365                         (this, TRANSFERINDEX, nextIndex,
2366                                 nextBound = (nextIndex > stride ?

2368                     bound = nextBound;

3113      * serves as a base class for other iterators and spliterators.
3114      *
3115      * Method advance visits once each still-valid node that was
3116      * reachable upon iterator construction. It might miss some that
3117      * were added to a bin after the bin was visited, which is OK wrt
3118      * consistency guarantees. Maintaining this property in the face
3119      * of possible ongoing resizes requires a fair amount of
3120      * bookkeeping state that is difficult to optimize away amidst
3121      * volatile accesses.  Even so, traversal maintains reasonable
3122      * throughput.
3123      *
3124      * Normally, iteration proceeds bin-by-bin traversing lists.
3125      * However, if the table has been resized, then all future steps
3126      * must traverse both the bin at the current index as well as at
3127      * (index + baseSize); and so on for further resizings. To
3128      * paranoically cope with potential sharing by users of iterators
3129      * across threads, iteration terminates if a bounds checks fails
3130      * for a table read.
3131      */

3134         Node<K,V> next;         // the next entry to use

3137         int baseLimit;          // index bound for initial table

3150          */

3152             Node<K,V> e;
3153             if ((e = next) != null)
3154                 e = e.next;
3155             for (;;) {

3157                 if (e != null)
3158                     return next = e;
3159                 if (baseIndex >= baseLimit || (t = tab) == null ||
3160                         (n = t.length) <= (i = index) || i < 0)
3161                     return next = null;
3162                 if ((e = tabAt(t, index)) != null && e.hash < 0) {
3163                     if (e instanceof ForwardingNode) {
3164                         tab = ((ForwardingNode<K,V>)e).nextTable;
3165                         e = null;

3167                     }

3169                         e = ((TreeBin<K,V>)e).first;
3170                     else
3171                         e = null;
3172                 }

6051     private final void fullAddCount(long x, CounterHashCode hc,
6052                                     boolean wasUncontended) {
6053         int h;
6054         if (hc == null) {
6055             hc = new CounterHashCode();
6056             int s = counterHashCodeGenerator.addAndGet(SEED_INCREMENT);
6057             h = hc.code = (s == 0) ? 1 : s; // Avoid zero
6058             threadCounterHashCode.set(hc);
6059         }
6060         else
6061             h = hc.code;
6062         boolean collide = false;                // True if last slot nonempty
6063         for (;;) {
6064             CounterCell[] as; CounterCell a; int n; long v;
6065             if ((as = counterCells) != null && (n = as.length) > 0) {
6066                 if ((a = as[(n - 1) & h]) == null) {
6067                     if (cellsBusy == 0) {            // Try to attach new Cell
6068                         CounterCell r = new CounterCell(x); // Optimistic create
6069                         if (cellsBusy == 0 &&
6070                                 U.compareAndSwapInt(this, CELLSBUSY, 0, 1)) {
6071                             boolean created = false;
6072                             try {               // Recheck under lock
6073                                 CounterCell[] rs; int m, j;
6074                                 if ((rs = counterCells) != null &&
6075                                         (m = rs.length) > 0 &&
6076                                         rs[j = (m - 1) & h] == null) {
6077                                     rs[j] = r;
6078                                     created = true;
6079                                 }
6080                             } finally {
6081                                 cellsBusy = 0;
6082                             }
6083                             if (created)
6084                                 break;
6085                             continue;           // Slot is now non-empty
6086                         }
6087                     }
6088                     collide = false;
6089                 }
6090                 else if (!wasUncontended)       // CAS already known to fail
6091                     wasUncontended = true;      // Continue after rehash
6092                 else if (U.compareAndSwapLong(a, CELLVALUE, v = a.value, v + x))
6093                     break;
6094                 else if (counterCells != as || n >= NCPU)
6095                     collide = false;            // At max size or stale
6096                 else if (!collide)
6097                     collide = true;
6098                 else if (cellsBusy == 0 &&
6099                         U.compareAndSwapInt(this, CELLSBUSY, 0, 1)) {
6100                     try {
6101                         if (counterCells == as) {// Expand table unless stale
6102                             CounterCell[] rs = new CounterCell[n << 1];
6103                             for (int i = 0; i < n; ++i)
6104                                 rs[i] = as[i];
6105                             counterCells = rs;
6106                         }
6107                     } finally {
6108                         cellsBusy = 0;
6109                     }
6110                     collide = false;
6111                     continue;                   // Retry with expanded table
6112                 }
6113                 h ^= h << 13;                   // Rehash
6114                 h ^= h >>> 17;
6115                 h ^= h << 5;
6116             }
6117             else if (cellsBusy == 0 && counterCells == as &&
6118                     U.compareAndSwapInt(this, CELLSBUSY, 0, 1)) {
6119                 boolean init = false;
6120                 try {                           // Initialize table
6121                     if (counterCells == as) {
6122                         CounterCell[] rs = new CounterCell[2];
6123                         rs[h & 1] = new CounterCell(x);
6124                         counterCells = rs;
6125                         init = true;
6126                     }
6127                 } finally {
6128                     cellsBusy = 0;
6129                 }
6130                 if (init)
6131                     break;
6132             }
6133             else if (U.compareAndSwapLong(this, BASECOUNT, v = baseCount, v + x))
6134                 break;                          // Fall back on using base
6135         }
6136         hc.code = h;                            // Record index for next time
