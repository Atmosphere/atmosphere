40  /**
41   * An {@link ExecutorService} for running {@link ForkJoinTask}s.
42   * A {@code ForkJoinPool} provides the entry point for submissions
43   * from non-{@code ForkJoinTask} clients, as well as management and
44   * monitoring operations.
45   *
46   * <p>A {@code ForkJoinPool} differs from other kinds of {@link
47   * ExecutorService} mainly by virtue of employing
48   * <em>work-stealing</em>: all threads in the pool attempt to find and
49   * execute tasks submitted to the pool and/or created by other active
50   * tasks (eventually blocking waiting for work if none exist). This
51   * enables efficient processing when most tasks spawn other subtasks
52   * (as do most {@code ForkJoinTask}s), as well as when many small
53   * tasks are submitted to the pool from external clients.  Especially
54   * when setting <em>asyncMode</em> to true in constructors, {@code
55   * ForkJoinPool}s may also be appropriate for use with event-style
56   * tasks that are never joined.
57   *
58   * <p>A static {@link #commonPool()} is available and appropriate for
59   * most applications. The common pool is used by any ForkJoinTask that
60   * is not explicitly submitted to a specified pool. Using the common
61   * pool normally reduces resource usage (its threads are slowly
62   * reclaimed during periods of non-use, and reinstated upon subsequent
63   * use).
64   *
65   * <p>For applications that require separate or custom pools, a {@code
66   * ForkJoinPool} may be constructed with a given target parallelism
67   * level; by default, equal to the number of available processors. The
68   * pool attempts to maintain enough active (or available) threads by
69   * dynamically adding, suspending, or resuming internal worker
70   * threads, even if some tasks are stalled waiting to join others.
71   * However, no such adjustments are guaranteed in the face of blocked
72   * I/O or other unmanaged synchronization. The nested {@link
73   * ManagedBlocker} interface enables extension of the kinds of
74   * synchronization accommodated.
75   *
76   * <p>In addition to execution and lifecycle control methods, this
77   * class provides status check methods (for example
78   * {@link #getStealCount}) that are intended to aid in developing,
79   * tuning, and monitoring fork/join applications. Also, method
80   * {@link #toString} returns indications of pool state in a
81   * convenient form for informal monitoring.
82   *
83   * <p>As is the case with other ExecutorServices, there are three
84   * main task execution methods summarized in the following table.
85   * These are designed to be used primarily by clients not already
86   * engaged in fork/join computations in the current pool.  The main
87   * forms of these methods accept instances of {@code ForkJoinTask},
88   * but overloaded forms also allow mixed execution of plain {@code
89   * Runnable}- or {@code Callable}- based activities as well.  However,
90   * tasks that are already executing in a pool should normally instead
91   * use the within-computation forms listed in the table unless using
92   * async event-style tasks that are not usually joined, in which case
93   * there is little difference among choice of methods.
94   *
95   * <table BORDER CELLPADDING=3 CELLSPACING=1>
96   * <caption>Summary of task execution methods</caption>
97   *  <tr>
98   *    <td></td>
99   *    <td ALIGN=CENTER> <b>Call from non-fork/join clients</b></td>
100  *    <td ALIGN=CENTER> <b>Call from within fork/join computations</b></td>
101  *  </tr>
102  *  <tr>
103  *    <td> <b>Arrange async execution</b></td>
104  *    <td> {@link #execute(ForkJoinTask)}</td>
105  *    <td> {@link ForkJoinTask#fork}</td>
106  *  </tr>
107  *  <tr>
108  *    <td> <b>Await and obtain result</b></td>
109  *    <td> {@link #invoke(ForkJoinTask)}</td>
110  *    <td> {@link ForkJoinTask#invoke}</td>
111  *  </tr>
112  *  <tr>
113  *    <td> <b>Arrange exec and obtain Future</b></td>
114  *    <td> {@link #submit(ForkJoinTask)}</td>
115  *    <td> {@link ForkJoinTask#fork} (ForkJoinTasks <em>are</em> Futures)</td>
116  *  </tr>
117  * </table>
118  *
119  * <p>The common pool is by default constructed with default
120  * parameters, but these may be controlled by setting three
121  * {@linkplain System#getProperty system properties}:
122  * <ul>
123  * <li>{@code java.util.concurrent.ForkJoinPool.common.parallelism}
124  * - the parallelism level, a non-negative integer
125  * <li>{@code java.util.concurrent.ForkJoinPool.common.threadFactory}
126  * - the class name of a {@link ForkJoinWorkerThreadFactory}
127  * <li>{@code java.util.concurrent.ForkJoinPool.common.exceptionHandler}
128  * - the class name of a {@link UncaughtExceptionHandler}
129  * </ul>
130  * The system class loader is used to load these classes.
131  * Upon any error in establishing these settings, default parameters
132  * are used. It is possible to disable or limit the use of threads in
133  * the common pool by setting the parallelism property to zero, and/or
134  * using a factory that may return {@code null}.
135  *
136  * <p><b>Implementation notes</b>: This implementation restricts the
137  * maximum number of running threads to 32767. Attempts to create
138  * pools with greater than the maximum number result in
139  * {@code IllegalArgumentException}.
140  *
141  * <p>This implementation rejects submitted tasks (that is, by throwing
142  * {@link RejectedExecutionException}) only when the pool is shut down
143  * or internal resources have been exhausted.
144  *
145  * @since 1.7
146  * @author Doug Lea
147  */

151     /*
152      * Implementation Overview
153      *
154      * This class and its nested classes provide the main
155      * functionality and control for a set of worker threads:
156      * Submissions from non-FJ threads enter into submission queues.
157      * Workers take these tasks and typically split them into subtasks
158      * that may be stolen by other workers.  Preference rules give
159      * first priority to processing tasks from their own queues (LIFO
160      * or FIFO, depending on mode), then to randomized FIFO steals of
161      * tasks in other queues.
162      *
163      * WorkQueues
164      * ==========
165      *
166      * Most operations occur within work-stealing queues (in nested
167      * class WorkQueue).  These are special forms of Deques that
168      * support only three of the four possible end-operations -- push,
169      * pop, and poll (aka steal), under the further constraints that
170      * push and pop are called only from the owning thread (or, as
171      * extended here, under a lock), while poll may be called from
172      * other threads.  (If you are unfamiliar with them, you probably
173      * want to read Herlihy and Shavit's book "The Art of
174      * Multiprocessor programming", chapter 16 describing these in
175      * more detail before proceeding.)  The main work-stealing queue
176      * design is roughly similar to those in the papers "Dynamic
177      * Circular Work-Stealing Deque" by Chase and Lev, SPAA 2005
178      * (http://research.sun.com/scalable/pubs/index.html) and
179      * "Idempotent work stealing" by Michael, Saraswat, and Vechev,
180      * PPoPP 2009 (http://portal.acm.org/citation.cfm?id=1504186).
181      * See also "Correct and Efficient Work-Stealing for Weak Memory
182      * Models" by Le, Pop, Cohen, and Nardelli, PPoPP 2013
183      * (http://www.di.ens.fr/~zappa/readings/ppopp13.pdf) for an
184      * analysis of memory ordering (atomic, volatile etc) issues.  The
185      * main differences ultimately stem from GC requirements that we
186      * null out taken slots as soon as we can, to maintain as small a
187      * footprint as possible even in programs generating huge numbers
188      * of tasks. To accomplish this, we shift the CAS arbitrating pop
189      * vs poll (steal) from being on the indices ("base" and "top") to
190      * the slots themselves.  So, both a successful pop and poll
191      * mainly entail a CAS of a slot from non-null to null.  Because
192      * we rely on CASes of references, we do not need tag bits on base
193      * or top.  They are simple ints as used in any circular
194      * array-based queue (see for example ArrayDeque).  Updates to the
195      * indices must still be ordered in a way that guarantees that top
196      * == base means the queue is empty, but otherwise may err on the
197      * side of possibly making the queue appear nonempty when a push,
198      * pop, or poll have not fully committed. Note that this means
199      * that the poll operation, considered individually, is not
200      * wait-free. One thief cannot successfully continue until another
201      * in-progress one (or, if previously empty, a push) completes.
202      * However, in the aggregate, we ensure at least probabilistic
203      * non-blockingness.  If an attempted steal fails, a thief always
204      * chooses a different random victim target to try next. So, in
205      * order for one thief to progress, it suffices for any
206      * in-progress poll or new push on any empty queue to
207      * complete. (This is why we normally use method pollAt and its
208      * variants that try once at the apparent base index, else
209      * consider alternative actions, rather than method poll.)
210      *
211      * This approach also enables support of a user mode in which local
212      * task processing is in FIFO, not LIFO order, simply by using
213      * poll rather than pop.  This can be useful in message-passing
214      * frameworks in which tasks are never joined.  However neither
215      * mode considers affinities, loads, cache localities, etc, so
216      * rarely provide the best possible performance on a given
217      * machine, but portably provide good throughput by averaging over
218      * these factors.  (Further, even if we did try to use such
219      * information, we do not usually have a basis for exploiting it.
220      * For example, some sets of tasks profit from cache affinities,
221      * but others are harmed by cache pollution effects.)
222      *
223      * WorkQueues are also used in a similar way for tasks submitted
224      * to the pool. We cannot mix these tasks in the same queues used
225      * for work-stealing (this would contaminate lifo/fifo
226      * processing). Instead, we randomly associate submission queues
227      * with submitting threads, using a form of hashing.  The
228      * Submitter probe value serves as a hash code for
229      * choosing existing queues, and may be randomly repositioned upon
230      * contention with other submitters.  In essence, submitters act
231      * like workers except that they are restricted to executing local
232      * tasks that they submitted (or in the case of CountedCompleters,
233      * others with the same root task).  However, because most
234      * shared/external queue operations are more expensive than
235      * internal, and because, at steady state, external submitters
236      * will compete for CPU with workers, ForkJoinTask.join and
237      * related methods disable them from repeatedly helping to process
238      * tasks if all workers are active.  Insertion of tasks in shared
239      * mode requires a lock (mainly to protect in the case of
240      * resizing) but we use only a simple spinlock (using bits in
241      * field qlock), because submitters encountering a busy queue move
242      * on to try or create other queues -- they block only when
243      * creating and registering new queues.
244      *
245      * Management
246      * ==========
247      *
248      * The main throughput advantages of work-stealing stem from
249      * decentralized control -- workers mostly take tasks from
250      * themselves or each other. We cannot negate this in the
251      * implementation of other management responsibilities. The main
252      * tactic for avoiding bottlenecks is packing nearly all
253      * essentially atomic control state into two volatile variables
254      * that are by far most often read (not written) as status and
255      * consistency checks.
256      *
257      * Field "ctl" contains 64 bits holding all the information needed
258      * to atomically decide to add, inactivate, enqueue (on an event
259      * queue), dequeue, and/or re-activate workers.  To enable this
260      * packing, we restrict maximum parallelism to (1<<15)-1 (which is
261      * far in excess of normal operating range) to allow ids, counts,
262      * and their negations (used for thresholding) to fit into 16bit
263      * fields.
264      *
265      * Field "plock" is a form of sequence lock with a saturating
266      * shutdown bit (similarly for per-queue "qlocks"), mainly
267      * protecting updates to the workQueues array, as well as to
268      * enable shutdown.  When used as a lock, it is normally only very
269      * briefly held, so is nearly always available after at most a
270      * brief spin, but we use a monitor-based backup strategy to
271      * block when needed.
272      *
273      * Recording WorkQueues.  WorkQueues are recorded in the
274      * "workQueues" array that is created upon first use and expanded
275      * if necessary.  Updates to the array while recording new workers
276      * and unrecording terminated ones are protected from each other
277      * by a lock but the array is otherwise concurrently readable, and
278      * accessed directly.  To simplify index-based operations, the
279      * array size is always a power of two, and all readers must
280      * tolerate null slots. Worker queues are at odd indices. Shared
281      * (submission) queues are at even indices, up to a maximum of 64
282      * slots, to limit growth even if array needs to expand to add
283      * more workers. Grouping them together in this way simplifies and
284      * speeds up task scanning.
285      *
286      * All worker thread creation is on-demand, triggered by task
287      * submissions, replacement of terminated workers, and/or
288      * compensation for blocked workers. However, all other support
289      * code is set up to work with other policies.  To ensure that we
290      * do not hold on to worker references that would prevent GC, ALL
291      * accesses to workQueues are via indices into the workQueues
292      * array (which is one source of some of the messy code
293      * constructions here). In essence, the workQueues array serves as
294      * a weak reference mechanism. Thus for example the wait queue
295      * field of ctl stores indices, not references.  Access to the
296      * workQueues in associated methods (for example signalWork) must
297      * both index-check and null-check the IDs. All such accesses
298      * ignore bad IDs by returning out early from what they are doing,
299      * since this can only be associated with termination, in which
300      * case it is OK to give up.  All uses of the workQueues array
301      * also check that it is non-null (even if previously
302      * non-null). This allows nulling during termination, which is
303      * currently not necessary, but remains an option for
304      * resource-revocation-based shutdown schemes. It also helps
305      * reduce JIT issuance of uncommon-trap code, which tends to
306      * unnecessarily complicate control flow in some methods.
307      *
308      * Event Queuing. Unlike HPC work-stealing frameworks, we cannot
309      * let workers spin indefinitely scanning for tasks when none can
310      * be found immediately, and we cannot start/resume workers unless
311      * there appear to be tasks available.  On the other hand, we must
312      * quickly prod them into action when new tasks are submitted or
313      * generated. In many usages, ramp-up time to activate workers is
314      * the main limiting factor in overall performance (this is
315      * compounded at program start-up by JIT compilation and
316      * allocation). So we try to streamline this as much as possible.
317      * We park/unpark workers after placing in an event wait queue
318      * when they cannot find work. This "queue" is actually a simple
319      * Treiber stack, headed by the "id" field of ctl, plus a 15bit
320      * counter value (that reflects the number of times a worker has
321      * been inactivated) to avoid ABA effects (we need only as many
322      * version numbers as worker threads). Successors are held in
323      * field WorkQueue.nextWait.  Queuing deals with several intrinsic
324      * races, mainly that a task-producing thread can miss seeing (and
325      * signalling) another thread that gave up looking for work but
326      * has not yet entered the wait queue. We solve this by requiring
327      * a full sweep of all workers (via repeated calls to method
328      * scan()) both before and after a newly waiting worker is added
329      * to the wait queue.  Because enqueued workers may actually be
330      * rescanning rather than waiting, we set and clear the "parker"
331      * field of WorkQueues to reduce unnecessary calls to unpark.
332      * (This requires a secondary recheck to avoid missed signals.)
333      * Note the unusual conventions about Thread.interrupts
334      * surrounding parking and other blocking: Because interrupts are
335      * used solely to alert threads to check termination, which is
336      * checked anyway upon blocking, we clear status (using
337      * Thread.interrupted) before any call to park, so that park does
338      * not immediately return due to status being set via some other
339      * unrelated call to interrupt in user code.
340      *
341      * Signalling.  We create or wake up workers only when there
342      * appears to be at least one task they might be able to find and
343      * execute.  When a submission is added or another worker adds a
344      * task to a queue that has fewer than two tasks, they signal
345      * waiting workers (or trigger creation of new ones if fewer than
346      * the given parallelism level -- signalWork).  These primary
347      * signals are buttressed by others whenever other threads remove
348      * a task from a queue and notice that there are other tasks there
349      * as well.  So in general, pools will be over-signalled. On most
350      * platforms, signalling (unpark) overhead time is noticeably
351      * long, and the time between signalling a thread and it actually
352      * making progress can be very noticeably long, so it is worth
353      * offloading these delays from critical paths as much as
354      * possible. Additionally, workers spin-down gradually, by staying
355      * alive so long as they see the ctl state changing.  Similar
356      * stability-sensing techniques are also used before blocking in
357      * awaitJoin and helpComplete.
358      *
359      * Trimming workers. To release resources after periods of lack of
360      * use, a worker starting to wait when the pool is quiescent will
361      * time out and terminate if the pool has remained quiescent for a
362      * given period -- a short period if there are more threads than
363      * parallelism, longer as the number of threads decreases. This
364      * will slowly propagate, eventually terminating all workers after
365      * periods of non-use.
366      *
367      * Shutdown and Termination. A call to shutdownNow atomically sets
368      * a plock bit and then (non-atomically) sets each worker's
369      * qlock status, cancels all unprocessed tasks, and wakes up
370      * all waiting workers.  Detecting whether termination should
371      * commence after a non-abrupt shutdown() call requires more work
372      * and bookkeeping. We need consensus about quiescence (i.e., that
373      * there is no more work). The active count provides a primary
374      * indication but non-abrupt shutdown still requires a rechecking
375      * scan for any workers that are inactive but not queued.
376      *
377      * Joining Tasks
378      * =============
379      *
380      * Any of several actions may be taken when one worker is waiting
381      * to join a task stolen (or always held) by another.  Because we
382      * are multiplexing many tasks on to a pool of workers, we can't
383      * just let them block (as in Thread.join).  We also cannot just
384      * reassign the joiner's run-time stack with another and replace
385      * it later, which would be a form of "continuation", that even if
386      * possible is not necessarily a good idea since we sometimes need
387      * both an unblocked task and its continuation to progress.
388      * Instead we combine two tactics:
389      *
390      *   Helping: Arranging for the joiner to execute some task that it
391      *      would be running if the steal had not occurred.
392      *
393      *   Compensating: Unless there are already enough live threads,
394      *      method tryCompensate() may create or re-activate a spare
395      *      thread to compensate for blocked joiners until they unblock.
396      *
397      * A third form (implemented in tryRemoveAndExec) amounts to
398      * helping a hypothetical compensator: If we can readily tell that
399      * a possible action of a compensator is to steal and execute the
400      * task being joined, the joining thread can do so directly,
401      * without the need for a compensation thread (although at the
402      * expense of larger run-time stacks, but the tradeoff is
403      * typically worthwhile).
404      *
405      * The ManagedBlocker extension API can't use helping so relies
406      * only on compensation in method awaitBlocker.
407      *
408      * The algorithm in tryHelpStealer entails a form of "linear"
409      * helping: Each worker records (in field currentSteal) the most
410      * recent task it stole from some other worker. Plus, it records
411      * (in field currentJoin) the task it is currently actively
412      * joining. Method tryHelpStealer uses these markers to try to
413      * find a worker to help (i.e., steal back a task from and execute
414      * it) that could hasten completion of the actively joined task.
415      * In essence, the joiner executes a task that would be on its own
416      * local deque had the to-be-joined task not been stolen. This may
417      * be seen as a conservative variant of the approach in Wagner &
418      * Calder "Leapfrogging: a portable technique for implementing
419      * efficient futures" SIGPLAN Notices, 1993
420      * (http://portal.acm.org/citation.cfm?id=155354). It differs in
421      * that: (1) We only maintain dependency links across workers upon
422      * steals, rather than use per-task bookkeeping.  This sometimes
423      * requires a linear scan of workQueues array to locate stealers,
424      * but often doesn't because stealers leave hints (that may become
425      * stale/wrong) of where to locate them.  It is only a hint
426      * because a worker might have had multiple steals and the hint
427      * records only one of them (usually the most current).  Hinting
428      * isolates cost to when it is needed, rather than adding to
429      * per-task overhead.  (2) It is "shallow", ignoring nesting and
430      * potentially cyclic mutual steals.  (3) It is intentionally
431      * racy: field currentJoin is updated only while actively joining,
432      * which means that we miss links in the chain during long-lived
433      * tasks, GC stalls etc (which is OK since blocking in such cases
434      * is usually a good idea).  (4) We bound the number of attempts
435      * to find work (see MAX_HELP) and fall back to suspending the
436      * worker and if necessary replacing it with another.
437      *
438      * Helping actions for CountedCompleters are much simpler: Method
439      * helpComplete can take and execute any task with the same root
440      * as the task being waited on. However, this still entails some
441      * traversal of completer chains, so is less efficient than using
442      * CountedCompleters without explicit joins.
443      *
444      * It is impossible to keep exactly the target parallelism number
445      * of threads running at any given time.  Determining the
446      * existence of conservatively safe helping targets, the
447      * availability of already-created spares, and the apparent need
448      * to create new spares are all racy, so we rely on multiple
449      * retries of each.  Compensation in the apparent absence of
450      * helping opportunities is challenging to control on JVMs, where
451      * GC and other activities can stall progress of tasks that in
452      * turn stall out many other dependent tasks, without us being
453      * able to determine whether they will ever require compensation.
454      * Even though work-stealing otherwise encounters little
455      * degradation in the presence of more threads than cores,
456      * aggressively adding new threads in such cases entails risk of
457      * unwanted positive feedback control loops in which more threads
458      * cause more dependent stalls (as well as delayed progress of
459      * unblocked threads to the point that we know they are available)
460      * leading to more situations requiring more threads, and so
461      * on. This aspect of control can be seen as an (analytically
462      * intractable) game with an opponent that may choose the worst
463      * (for us) active thread to stall at any time.  We take several
464      * precautions to bound losses (and thus bound gains), mainly in
465      * methods tryCompensate and awaitJoin.
466      *
467      * Common Pool
468      * ===========
469      *
470      * The static common pool always exists after static
471      * initialization.  Since it (or any other created pool) need
472      * never be used, we minimize initial construction overhead and
473      * footprint to the setup of about a dozen fields, with no nested
474      * allocation. Most bootstrapping occurs within method
475      * fullExternalPush during the first submission to the pool.
476      *
477      * When external threads submit to the common pool, they can
478      * perform subtask processing (see externalHelpJoin and related
479      * methods).  This caller-helps policy makes it sensible to set
480      * common pool parallelism level to one (or more) less than the
481      * total number of available cores, or even zero for pure
482      * caller-runs.  We do not need to record whether external
483      * submissions are to the common pool -- if not, externalHelpJoin
484      * returns quickly (at the most helping to signal some common pool
485      * workers). These submitters would otherwise be blocked waiting
486      * for completion, so the extra effort (with liberally sprinkled
487      * task status checks) in inapplicable cases amounts to an odd
488      * form of limited spin-wait before blocking in ForkJoinTask.join.
489      *
490      * Style notes
491      * ===========
492      *
493      * There is a lot of representation-level coupling among classes
494      * ForkJoinPool, ForkJoinWorkerThread, and ForkJoinTask.  The
495      * fields of WorkQueue maintain data structures managed by
496      * ForkJoinPool, so are directly accessed.  There is little point
497      * trying to reduce this, since any associated future changes in
498      * representations will need to be accompanied by algorithmic
499      * changes anyway. Several methods intrinsically sprawl because
500      * they must accumulate sets of consistent reads of volatiles held
501      * in local variables.  Methods signalWork() and scan() are the
502      * main bottlenecks, so are especially heavily
503      * micro-optimized/mangled.  There are lots of inline assignments
504      * (of form "while ((local = field) != 0)") which are usually the
505      * simplest way to ensure the required read orderings (which are
506      * sometimes critical). This leads to a "C"-like style of listing
507      * declarations of these locals at the heads of methods or blocks.
508      * There are several occurrences of the unusual "do {} while
509      * (!cas...)"  which is the simplest way to force an update of a
510      * CAS'ed variable. There are also other coding oddities (including
511      * several unnecessary-looking hoisted null checks) that help
512      * some methods perform reasonably even when interpreted (not
513      * compiled).
514      *
515      * The order of declarations in this file is:

582     /**
583      * Queues supporting work-stealing as well as external task
584      * submission. See above for main rationale and algorithms.
585      * Implementation relies heavily on "Unsafe" intrinsics
586      * and selective use of "volatile":
587      *
588      * Field "base" is the index (mod array.length) of the least valid
589      * queue slot, which is always the next position to steal (poll)
590      * from if nonempty. Reads and writes require volatile orderings
591      * but not CAS, because updates are only performed after slot
592      * CASes.
593      *
594      * Field "top" is the index (mod array.length) of the next queue
595      * slot to push to or pop from. It is written only by owner thread
596      * for push, or under lock for external/shared push, and accessed
597      * by other threads only after reading (volatile) base.  Both top
598      * and base are allowed to wrap around on overflow, but (top -
599      * base) (or more commonly -(base - top) to force volatile read of
600      * base before top) still estimates size. The lock ("qlock") is
601      * forced to -1 on termination, causing all further lock attempts
602      * to fail. (Note: we don't need CAS for termination state because
603      * upon pool shutdown, all shared-queues will stop being used
604      * anyway.)  Nearly all lock bodies are set up so that exceptions
605      * within lock bodies are "impossible" (modulo JVM errors that
606      * would cause failure anyway.)
607      *
608      * The array slots are read and written using the emulation of
609      * volatiles/atomics provided by Unsafe. Insertions must in
610      * general use putOrderedObject as a form of releasing store to
611      * ensure that all writes to the task object are ordered before
612      * its publication in the queue.  All removals entail a CAS to
613      * null.  The array is always a power of two. To ensure safety of
614      * Unsafe array operations, all accesses perform explicit null
615      * checks and implicit bounds checks via power-of-two masking.
616      *
617      * In addition to basic queuing support, this class contains
618      * fields described elsewhere to control execution. It turns out
619      * to work better memory-layout-wise to include them in this class
620      * rather than a separate class.
621      *
622      * Performance on most platforms is very sensitive to placement of
623      * instances of both WorkQueues and their arrays -- we absolutely
624      * do not want multiple WorkQueue instances or multiple queue
625      * arrays sharing cache lines. (It would be best for queue objects
626      * and their arrays to share, but there is nothing available to
627      * help arrange that). The @Contended annotation alerts JVMs to
628      * try to keep instances apart.
629      */

634          * reduce or eliminate cacheline sharing among queues.

636          * the fact that JVMs often place arrays in locations that
637          * share GC bookkeeping (especially cardmarks) such that
638          * per-write accesses encounter serious memory contention.
639          */
640         static final int INITIAL_QUEUE_CAPACITY = 1 << 13;

646          * value a bit less than this to help users trap runaway
647          * programs before saturating systems.

708          * Pushes a task. Call only by owner in unshared queues.  (The
709          * shared-queue version is embedded in method externalPush.)
710          *
711          * @param task the task. Caller must ensure non-null.
712          * @throws RejectedExecutionException if array cannot be resized
713          */
714         final void push(ForkJoinTask<?> task) {
715             ForkJoinTask<?>[] a; ForkJoinPool p;
716             int s = top, n;
717             if ((a = array) != null) {    // ignore if queue removed
718                 int m = a.length - 1;
719                 U.putOrderedObject(a, ((m & s) << ASHIFT) + ABASE, task);
720                 if ((n = (top = s + 1) - base) <= 2)
721                     (p = pool).signalWork(p.workQueues, this);
722                 else if (n >= m)

729          * by owner or with lock held -- it is OK for base, but not
730          * top, to move while resizings are in progress.
731          */
732         final ForkJoinTask<?>[] growArray() {
733             ForkJoinTask<?>[] oldA = array;
734             int size = oldA != null ? oldA.length << 1 : INITIAL_QUEUE_CAPACITY;
735             if (size > MAXIMUM_QUEUE_CAPACITY)
736                 throw new RejectedExecutionException("Queue capacity exceeded");
737             int oldMask, t, b;
738             ForkJoinTask<?>[] a = array = new ForkJoinTask<?>[size];
739             if (oldA != null && (oldMask = oldA.length - 1) >= 0 &&
740                     (t = top) - (b = base) > 0) {
741                 int mask = size - 1;
742                 do {
743                     ForkJoinTask<?> x;
744                     int oldj = ((b & oldMask) << ASHIFT) + ABASE;
745                     int j    = ((b &    mask) << ASHIFT) + ABASE;

747                     if (x != null &&

761             if ((a = array) != null && (m = a.length - 1) >= 0) {
762                 for (int s; (s = top - 1) - base >= 0;) {
763                     long j = ((m & s) << ASHIFT) + ABASE;
764                     if ((t = (ForkJoinTask<?>)U.getObject(a, j)) == null)
765                         break;
766                     if (U.compareAndSwapObject(a, j, t, null)) {

794          * Takes next task, if one exists, in FIFO order.
795          */
796         final ForkJoinTask<?> poll() {
797             ForkJoinTask<?>[] a; int b; ForkJoinTask<?> t;
798             while ((b = base) - top < 0 && (a = array) != null) {
799                 int j = (((a.length - 1) & b) << ASHIFT) + ABASE;
800                 t = (ForkJoinTask<?>)U.getObjectVolatile(a, j);
801                 if (t != null) {
802                     if (U.compareAndSwapObject(a, j, t, null)) {
803                         U.putOrderedInt(this, QBASE, b + 1);
804                         return t;
805                     }
806                 }
807                 else if (base == b) {
808                     if (b + 1 == top)
809                         break;
810                     Thread.yield(); // wait for lagging update (very rare)
811                 }
812             }
813             return null;
814         }

874         final void runTask(ForkJoinTask<?> task) {
875             if ((currentSteal = task) != null) {
876                 task.doExec();

878                 int md = mode;
879                 ++nsteals;
880                 currentSteal = null;

888                         if (t == null)

890                         if (U.compareAndSwapObject(a, i, t, null)) {

899         /**
900          * If present, removes from queue and executes the given task,
901          * or any other cancelled task. Returns (true) on any CAS
902          * or consistency check failure so caller can retry.
903          *
904          * @return false if no progress can be made, else true
905          */
906         final boolean tryRemoveAndExec(ForkJoinTask<?> task) {
907             boolean stat;
908             ForkJoinTask<?>[] a; int m, s, b, n;
909             if (task != null && (a = array) != null && (m = a.length - 1) >= 0 &&
910                     (n = (s = top) - (b = base)) > 0) {
911                 boolean removed = false, empty = true;
912                 stat = true;
913                 for (ForkJoinTask<?> t;;) {           // traverse from s to b
914                     long j = ((--s & m) << ASHIFT) + ABASE;
915                     t = (ForkJoinTask<?>)U.getObject(a, j);
916                     if (t == null)                    // inconsistent length
917                         break;
918                     else if (t == task) {
919                         if (s + 1 == top) {           // pop
920                             if (!U.compareAndSwapObject(a, j, task, null))
921                                 break;
922                             top = s;
923                             removed = true;
924                         }
925                         else if (base == b)           // replace with proxy
926                             removed = U.compareAndSwapObject(a, j, task,
927                                     new EmptyTask());
928                         break;
929                     }
930                     else if (t.status >= 0)
931                         empty = false;
932                     else if (s + 1 == top) {          // pop and throw away
933                         if (U.compareAndSwapObject(a, j, t, null))
934                             top = s;
935                         break;
936                     }
937                     if (--n == 0) {
938                         if (!empty && base == b)
939                             stat = false;
940                         break;
941                     }
942                 }
943                 if (removed)
944                     task.doExec();
945             }
946             else
947                 stat = false;
948             return stat;
949         }

1163     /*
1164      * Bits and masks for control variables
1165      *
1166      * Field ctl is a long packed with:
1167      * AC: Number of active running workers minus target parallelism (16 bits)
1168      * TC: Number of total workers minus target parallelism (16 bits)
1169      * ST: true if pool is terminating (1 bit)
1170      * EC: the wait count of top waiting thread (15 bits)
1171      * ID: poolIndex of top of Treiber stack of waiters (16 bits)
1172      *
1173      * When convenient, we can extract the upper 32 bits of counts and
1174      * the lower 32 bits of queue state, u = (int)(ctl >>> 32) and e =
1175      * (int)ctl.  The ec field is never accessed alone, but always
1176      * together with id and st. The offsets of counts by the target
1177      * parallelism and the positionings of fields makes it possible to
1178      * perform the most common checks via sign tests of fields: When
1179      * ac is negative, there are not enough active workers, when tc is
1180      * negative, there are not enough total workers, and when e is
1181      * negative, the pool is terminating.  To deal with these possibly
1182      * negative fields, we use casts in and out of "short" and/or
1183      * signed shifts to maintain signedness.
1184      *
1185      * When a thread is queued (inactivated), its eventCount field is
1186      * set negative, which is the only way to tell if a worker is
1187      * prevented from executing tasks, even though it must continue to
1188      * scan for them to avoid queuing races. Note however that
1189      * eventCount updates lag releases so usage requires care.
1190      *
1191      * Field plock is an int packed with:
1192      * SHUTDOWN: true if shutdown is enabled (1 bit)
1193      * SEQ:  a sequence lock, with PL_LOCK bit set if locked (30 bits)
1194      * SIGNAL: set when threads may be waiting on the lock (1 bit)
1195      *
1196      * The sequence number enables simple consistency checks:
1197      * Staleness of read-only operations on the workQueues array can
1198      * be checked by comparing plock before vs after the reads.
1199      */

1265     /**
1266      * Acquires the plock lock to protect worker array and related
1267      * updates. This method is called only if an initial CAS on plock
1268      * fails. This acts as a spinlock for normal cases, but falls back
1269      * to builtin monitor to block when (rarely) needed. This would be
1270      * a terrible idea for a highly contended lock, but works fine as
1271      * a more conservative alternative to a pure spinlock.
1272      */
1273     private int acquirePlock() {
1274         int spins = PL_SPINS, ps, nps;
1275         for (;;) {
1276             if (((ps = plock) & PL_LOCK) == 0 &&
1277                     U.compareAndSwapInt(this, PLOCK, ps, nps = ps + PL_LOCK))
1278                 return nps;
1279             else if (spins >= 0) {
1280                 if (ThreadLocalRandom.current().nextInt() >= 0)
1281                     --spins;
1282             }
1283             else if (U.compareAndSwapInt(this, PLOCK, ps, ps | PL_SIGNAL)) {
1284                 synchronized (this) {
1285                     if ((plock & PL_SIGNAL) != 0) {
1286                         try {
1287                             wait();
1288                         } catch (InterruptedException ie) {
1289                             try {
1290                                 Thread.currentThread().interrupt();
1291                             } catch (SecurityException ignore) {
1292                             }
1293                         }
1294                     }
1295                     else
1296                         notifyAll();
1297                 }
1298             }
1299         }
1300     }

1342     /**
1343      * Callback from ForkJoinWorkerThread to establish and record its
1344      * WorkQueue. To avoid scanning bias due to packing entries in
1345      * front of the workQueues array, we treat the array as a simple
1346      * power-of-two hash table using per-thread seed as hash,
1347      * expanding as needed.
1348      *
1349      * @param wt the worker thread
1350      * @return the worker's queue
1351      */
1352     final WorkQueue registerWorker(ForkJoinWorkerThread wt) {
1353         UncaughtExceptionHandler handler; WorkQueue[] ws; int s, ps;
1354         wt.setDaemon(true);
1355         if ((handler = ueh) != null)
1356             wt.setUncaughtExceptionHandler(handler);
1357         do {} while (!U.compareAndSwapInt(this, INDEXSEED, s = indexSeed,
1358                 s += SEED_INCREMENT) ||
1359                 s == 0); // skip 0
1360         WorkQueue w = new WorkQueue(this, wt, mode, s);
1361         if (((ps = plock) & PL_LOCK) != 0 ||
1362                 !U.compareAndSwapInt(this, PLOCK, ps, ps += PL_LOCK))
1363             ps = acquirePlock();
1364         int nps = (ps & SHUTDOWN) | ((ps + PL_LOCK) & ~SHUTDOWN);
1365         try {
1366             if ((ws = workQueues) != null) {    // skip if shutting down
1367                 int n = ws.length, m = n - 1;
1368                 int r = (s << 1) | 1;           // use odd-numbered indices
1369                 if (ws[r &= m] != null) {       // collision
1370                     int probes = 0;             // step by approx half size
1371                     int step = (n <= 4) ? 2 : ((n >>> 1) & EVENMASK) + 2;
1372                     while (ws[r = (r + step) & m] != null) {
1373                         if (++probes >= n) {
1374                             workQueues = ws = Arrays.copyOf(ws, n <<= 1);
1375                             m = n - 1;
1376                             probes = 0;
1377                         }
1378                     }
1379                 }
1380                 w.poolIndex = (short)r;
1381                 w.eventCount = r; // volatile write orders
1382                 ws[r] = w;
1383             }
1384         } finally {
1385             if (!U.compareAndSwapInt(this, PLOCK, ps, nps))
1386                 releasePlock(nps);
1387         }
1388         wt.setName(workerNamePrefix.concat(Integer.toString(w.poolIndex >>> 1)));
1389         return w;
1390     }

1394      * to construct or start a worker.  Removes record of worker from
1395      * array, and adjusts counts. If pool is shutting down, tries to
1396      * complete termination.
1397      *
1398      * @param wt the worker thread, or null if construction failed
1399      * @param ex the exception causing failure, or null if none

1402         WorkQueue w = null;
1403         if (wt != null && (w = wt.workQueue) != null) {
1404             int ps; long sc;
1405             w.qlock = -1;                // ensure set
1406             do {} while (!U.compareAndSwapLong(this, STEALCOUNT,
1407                     sc = stealCount,
1408                     sc + w.nsteals));
1409             if (((ps = plock) & PL_LOCK) != 0 ||
1410                     !U.compareAndSwapInt(this, PLOCK, ps, ps += PL_LOCK))
1411                 ps = acquirePlock();
1412             int nps = (ps & SHUTDOWN) | ((ps + PL_LOCK) & ~SHUTDOWN);
1413             try {
1414                 int idx = w.poolIndex;
1415                 WorkQueue[] ws = workQueues;
1416                 if (ws != null && idx >= 0 && idx < ws.length && ws[idx] == w)
1417                     ws[idx] = null;
1418             } finally {
1419                 if (!U.compareAndSwapInt(this, PLOCK, ps, nps))
1420                     releasePlock(nps);
1421             }
1422         }
1423 
1424         long c;                          // adjust ctl counts
1425         do {} while (!U.compareAndSwapLong
1426                 (this, CTL, c = ctl, (((c - AC_UNIT) & AC_MASK) |
1427                         ((c - TC_UNIT) & TC_MASK) |
1428                         (c & ~(AC_MASK|TC_MASK)))));
1429 
1430         if (!tryTerminate(false, false) && w != null && w.array != null) {
1431             w.cancelAll();               // cancel remaining tasks
1432             WorkQueue[] ws; WorkQueue v; Thread p; int u, i, e;
1433             while ((u = (int)((c = ctl) >>> 32)) < 0 && (e = (int)c) >= 0) {
1434                 if (e > 0) {             // activate or create replacement
1435                     if ((ws = workQueues) == null ||
1436                             (i = e & SMASK) >= ws.length ||
1437                             (v = ws[i]) == null)
1438                         break;
1439                     long nc = (((long)(v.nextWait & E_MASK)) |
1440                             ((long)(u + UAC_UNIT) << 32));
1441                     if (v.eventCount != (e | INT_SIGN))
1442                         break;
1443                     if (U.compareAndSwapLong(this, CTL, c, nc)) {
1444                         v.eventCount = (e + E_SEQ) & E_MASK;
1445                         if ((p = v.parker) != null)
1446                             U.unpark(p);
1447                         break;
1448                     }
1449                 }
1450                 else {
1451                     if ((short)u < 0)
1452                         tryAddWorker();
1453                     break;
1454                 }
1455             }
1456         }
1457         if (ex == null)                     // help clean refs on way out
1458             ForkJoinTask.helpExpungeStaleExceptions();

1460             ForkJoinTask.rethrow(ex);

1467      * holds only pseudo-random seed / index that is used to choose
1468      * submission queues in method externalPush. In the future, this may
1469      * also incorporate a means to implement different task rejection

1476      * unlikely to collide with previous values. Seeds are then
1477      * randomly modified upon collisions using xorshifts, which

1485     /**
1486      * Unless shutting down, adds the given task to a submission queue
1487      * at submitter's current queue index (modulo submission
1488      * range). Only the most common path is directly handled in this
1489      * method. All others are relayed to fullExternalPush.
1490      *
1491      * @param task the task. Caller must ensure non-null.
1492      */
1493     final void externalPush(ForkJoinTask<?> task) {
1494         Submitter z = submitters.get();

1496         int ps = plock;
1497         WorkQueue[] ws = workQueues;
1498         if (z != null && ps > 0 && ws != null && (m = (ws.length - 1)) >= 0 &&
1499                 (q = ws[m & (r = z.seed) & SQMASK]) != null && r != 0 &&
1500                 U.compareAndSwapInt(q, QLOCK, 0, 1)) { // lock
1501             if ((a = q.array) != null &&
1502                     (am = a.length - 1) > (n = (s = q.top) - q.base)) {
1503                 int j = ((am & s) << ASHIFT) + ABASE;
1504                 U.putOrderedObject(a, j, task);
1505                 q.top = s + 1;                     // push on to deque
1506                 q.qlock = 0;
1507                 if (n <= 1)
1508                     signalWork(ws, q);
1509                 return;
1510             }
1511             q.qlock = 0;
1512         }
1513         fullExternalPush(task);
1514     }

1516     /**
1517      * Full version of externalPush. This method is called, among
1518      * other times, upon the first submission of the first task to the
1519      * pool, so must perform secondary initialization.  It also
1520      * detects first submission by an external thread by looking up
1521      * its ThreadLocal, and creates a new shared queue if the one at
1522      * index if empty or contended. The plock lock body must be
1523      * exception-free (so no try/finally) so we optimistically
1524      * allocate new queues outside the lock and throw them away if
1525      * (very rarely) not needed.
1526      *
1527      * Secondary initialization occurs when plock is zero, to create
1528      * workQueue array and set plock to a valid value.  This lock body
1529      * must also be exception-free. Because the plock seq value can
1530      * eventually wrap around zero, this method harmlessly fails to
1531      * reinitialize if workQueues exists, while still advancing plock.
1532      */
1533     private void fullExternalPush(ForkJoinTask<?> task) {
1534         int r = 0; // random index seed
1535         for (Submitter z = submitters.get();;) {
1536             WorkQueue[] ws; WorkQueue q; int ps, m, k;
1537             if (z == null) {
1538                 if (U.compareAndSwapInt(this, INDEXSEED, r = indexSeed,
1539                         r += SEED_INCREMENT) && r != 0)
1540                     submitters.set(z = new Submitter(r));
1541             }
1542             else if (r == 0) {                  // move to a different index
1543                 r = z.seed;
1544                 r ^= r << 13;                   // same xorshift as WorkQueues
1545                 r ^= r >>> 17;
1546                 z.seed = r ^= (r << 5);
1547             }
1548             if ((ps = plock) < 0)
1549                 throw new RejectedExecutionException();
1550             else if (ps == 0 || (ws = workQueues) == null ||
1551                     (m = ws.length - 1) < 0) { // initialize workQueues
1552                 int p = parallelism;            // find power of two table size
1553                 int n = (p > 1) ? p - 1 : 1;    // ensure at least 2 slots
1554                 n |= n >>> 1; n |= n >>> 2;  n |= n >>> 4;
1555                 n |= n >>> 8; n |= n >>> 16; n = (n + 1) << 1;
1556                 WorkQueue[] nws = ((ws = workQueues) == null || ws.length == 0 ?
1557                         new WorkQueue[n] : null);
1558                 if (((ps = plock) & PL_LOCK) != 0 ||
1559                         !U.compareAndSwapInt(this, PLOCK, ps, ps += PL_LOCK))
1560                     ps = acquirePlock();
1561                 if (((ws = workQueues) == null || ws.length == 0) && nws != null)
1562                     workQueues = nws;
1563                 int nps = (ps & SHUTDOWN) | ((ps + PL_LOCK) & ~SHUTDOWN);
1564                 if (!U.compareAndSwapInt(this, PLOCK, ps, nps))
1565                     releasePlock(nps);
1566             }
1567             else if ((q = ws[k = r & m & SQMASK]) != null) {
1568                 if (q.qlock == 0 && U.compareAndSwapInt(q, QLOCK, 0, 1)) {
1569                     ForkJoinTask<?>[] a = q.array;
1570                     int s = q.top;
1571                     boolean submitted = false;
1572                     try {                      // locked version of push
1573                         if ((a != null && a.length > s + 1 - q.base) ||
1574                                 (a = q.growArray()) != null) {   // must presize
1575                             int j = (((a.length - 1) & s) << ASHIFT) + ABASE;
1576                             U.putOrderedObject(a, j, task);
1577                             q.top = s + 1;
1578                             submitted = true;
1579                         }
1580                     } finally {
1581                         q.qlock = 0;  // unlock
1582                     }
1583                     if (submitted) {
1584                         signalWork(ws, q);
1585                         return;
1586                     }
1587                 }
1588                 r = 0; // move on failure
1589             }
1590             else if (((ps = plock) & PL_LOCK) == 0) { // create new queue
1591                 q = new WorkQueue(this, null, SHARED_QUEUE, r);
1592                 q.poolIndex = (short)k;
1593                 if (((ps = plock) & PL_LOCK) != 0 ||
1594                         !U.compareAndSwapInt(this, PLOCK, ps, ps += PL_LOCK))
1595                     ps = acquirePlock();
1596                 if ((ws = workQueues) != null && k < ws.length && ws[k] == null)
1597                     ws[k] = q;
1598                 int nps = (ps & SHUTDOWN) | ((ps + PL_LOCK) & ~SHUTDOWN);
1599                 if (!U.compareAndSwapInt(this, PLOCK, ps, nps))
1600                     releasePlock(nps);
1601             }
1602             else
1603                 r = 0;
1604         }
1605     }

1665     /**
1666      * Scans for and, if found, runs one task, else possibly
1667      * inactivates the worker. This method operates on single reads of
1668      * volatile state and is designed to be re-invoked continuously,
1669      * in part because it returns upon detecting inconsistencies,
1670      * contention, or state changes that indicate possible success on
1671      * re-invocation.
1672      *
1673      * The scan searches for tasks across queues starting at a random
1674      * index, checking each at least twice.  The scan terminates upon
1675      * either finding a non-empty queue, or completing the sweep. If
1676      * the worker is not inactivated, it takes and runs a task from
1677      * this queue. Otherwise, if not activated, it tries to activate
1678      * itself or some other worker by signalling. On failure to find a
1679      * task, returns (for retry) if pool state may have changed during
1680      * an empty scan, or tries to inactivate if active, else possibly
1681      * blocks or terminates via method awaitWork.
1682      *
1683      * @param w the worker (via its WorkQueue)
1684      * @param r a random seed
1685      * @return worker qlock status if would have waited, else 0
1686      */
1687     private final int scan(WorkQueue w, int r) {
1688         WorkQueue[] ws; int m;
1689         long c = ctl;                            // for consistency check
1690         if ((ws = workQueues) != null && (m = ws.length - 1) >= 0 && w != null) {
1691             for (int j = m + m + 1, ec = w.eventCount;;) {
1692                 WorkQueue q; int b, e; ForkJoinTask<?>[] a; ForkJoinTask<?> t;
1693                 if ((q = ws[(r - j) & m]) != null &&
1694                         (b = q.base) - q.top < 0 && (a = q.array) != null) {
1695                     long i = (((a.length - 1) & b) << ASHIFT) + ABASE;

2149     /**
2150      * Returns a cheap heuristic guide for task partitioning when
2151      * programmers, frameworks, tools, or languages have little or no
2152      * idea about task granularity.  In essence by offering this
2153      * method, we ask users only about tradeoffs in overhead vs
2154      * expected throughput and its variance, rather than how finely to
2155      * partition tasks.
2156      *
2157      * In a steady state strict (tree-structured) computation, each
2158      * thread makes available for stealing enough tasks for other
2159      * threads to remain active. Inductively, if all threads play by
2160      * the same rules, each thread should make available only a
2161      * constant number of tasks.
2162      *
2163      * The minimum useful constant is just 1. But using a value of 1
2164      * would require immediate replenishment upon each steal to
2165      * maintain enough tasks, which is infeasible.  Further,
2166      * partitionings/granularities of offered tasks should minimize
2167      * steal rates, which in general means that threads nearer the top
2168      * of computation tree should generate more than those nearer the
2169      * bottom. In perfect steady state, each thread is at
2170      * approximately the same level of computation tree. However,
2171      * producing extra tasks amortizes the uncertainty of progress and
2172      * diffusion assumptions.
2173      *
2174      * So, users will want to use values larger (but not much larger)
2175      * than 1 to both smooth over transient shortages and hedge
2176      * against uneven progress; as traded off against the cost of
2177      * extra task overhead. We leave the user to pick a threshold
2178      * value to compare with the results of this call to guide
2179      * decisions, but recommend values such as 3.
2180      *
2181      * When all threads are active, it is on average OK to estimate
2182      * surplus strictly locally. In steady-state, if one thread is
2183      * maintaining say 2 surplus tasks, then so are others. So we can
2184      * just use estimated queue length.  However, this strategy alone
2185      * leads to serious mis-estimates in some non-steady-state
2186      * conditions (ramp-up, ramp-down, other stalls). We can detect
2187      * many of these by further considering the number of "idle"
2188      * threads, that are known to have zero queued tasks, so
2189      * compensate by a factor of (#idle/#active) threads.
2190      *
2191      * Note: The approximation of #busy workers as #active workers is
2192      * not very good under current signalling scheme, and should be
2193      * improved.

2202                     a > (p >>>= 1) ? 1 :

3114      * {@code isReleasable} must return {@code true} if blocking is

3123      * ensure sufficient parallelism.  Toward this end,
3124      * implementations of method {@code isReleasable} must be amenable
3125      * to repeated invocation.
